Step 3 of 6
Approval board
Build the approval pipeline for this ARB type. Requests flow through tiers in order. Each tier has its own group, approval pattern, and settings. Conditions can be created at any tier and carry forward — they do not block escalation.
ℹ
How tiers work. A request moves to the next tier only after the current tier approves. Any tier can reject — that ends the review. A tier configured to "approve and close" can stop the request there without escalating further. The final tier always closes.

4.2 — Auto-rejection rules
When a submission clearly violates your standards, the AI can return it to the submitter before it reaches any reviewer — with the specific reason, so they can fix and resubmit.
Enable auto-rejection for this ARB type
When a submission triggers any rule below, the AI returns it to the submitter with the specific reason — before it reaches any reviewer.
📖 How to write auto-rejection rules
Use clear, testable conditionsThe AI works best with rules it can check objectively. State the specific condition that must be true to reject.
Good examples✓ Mitigation plan is fewer than 100 characters✓ Expiry date exceeds 365 days from today✓ Business justification does not mention a specific standard being deviated from
Avoid✗ Poor quality submissions✗ Not good enough✗ Anything subjective without a measurable threshold
Keep the list short3–6 rules work best. Too many rules create false positives and undermine trust in automation.
Use for hard violations onlyAuto-rejection should be reserved for clear, unambiguous failures — not borderline cases that deserve human judgment.
One rule per lineWrite each rule on a new line. The AI evaluates each independently.
Auto-rejection rules — one per line
Write each rule as a plain-English condition. The AI will return any submission that triggers one or more rules.
⚠
Auto-rejection always takes priority over auto-approval. If a submission triggers any auto-rejection rule, it will be returned to the submitter even if all auto-approval conditions are also met.
4.3 — Auto-approval rules
When a submission clearly meets your standards, the AI can approve it without human review — so reviewers focus on requests that genuinely need their judgment.
Enable auto-approval for this ARB type
When a submission satisfies every condition below and no auto-rejection rules are triggered, the AI approves without routing to the board.
📖 How to write auto-approval conditions
Every condition must be metA submission is only auto-approved when ALL conditions are satisfied. Write each condition as something that must be true.
Good examples✓ Risk level is Low✓ Mitigation plan is present and more than 100 characters✓ Expiry date is within 90 days of the submission date✓ Deviation type is Product Defect or Vendor Constraint
Start conservativeBegin with 2–3 conditions that cover clear, routine cases. Expand coverage after reviewing how the AI handles real submissions. Mistaken auto-approvals are harder to recover from than unnecessary manual reviews.
Reference your document templateYou can reference sections you defined in Step 2 — e.g. "Mitigation approach section is present and describes specific actions, a timeline, and a named owner."
Reference uploaded standardsYou can reference documents uploaded in 4.1 — e.g. "Deviation type matches an approved exception category in NovaTech_Architecture_Standards_v4.pdf."
One condition per lineEach line is evaluated independently. All must be true for auto-approval to trigger.
Auto-approval conditions — one per line
All conditions must be satisfied for auto-approval. Auto-rejection rules always take priority — a submission that triggers any rejection rule will not be auto-approved.
⚠ Precedence: Auto-rejection always overrides auto-approval. If a submission triggers any rejection rule, it will be returned to the submitter regardless of whether all approval conditions are also met.

Step 6 of 6
Review & activate
Review your configuration before making this ARB type live. Click any section to jump back and make changes. Once activated, submitters can immediately raise requests of this type.
What will happen when this ARB type is active
1
Submitters will see Architecture Exception Request as an option in the Service Catalog when raising a new ARB request.
2
The submission form will collect 6 fields including Deviation type, Mitigation plan, and Expiry date, plus a 4-section document template covering Business context, Deviation details, Mitigation approach, and Security impact.
3
The AI will evaluate each submission against NovaTech_Architecture_Standards_v4.pdf. Submissions that trigger auto-rejection rules will be returned to the submitter with the specific reason before reaching any reviewer.
4
Requests that pass the AI check will be routed to Primary Reviewers → Domain Approvers → Architecture Board in sequence. Any tier can reject. The Architecture Board always closes.
5
Approved requests will generate a time-bound exception tracked against the submitter. Reminders at 60, 30, and 14 days before expiry. Extensions allowed up to 2 times within 365 days total.
6
Requests approved with conditions will have those conditions tracked automatically — reminders at 30 and 14 days, resolution comment required from the condition owner.
