Architecture exception request
Pre-configured for tracking temporary deviations from architecture standards.
Includes: Deviation type, Mitigation plan, Expiry date · Auto-approval for low-risk + documented mitigation · Lifecycle Active → Expired → Resolved → Closed
Best for: Technical debt, vendor workarounds, temporary exceptions with time limits

Pre-configured for reviewing new solutions, integrations, and technology adoption.

Step 3 of 7
Define the approval board
Build your approval pipeline. Each tier is a named group with a clear purpose — requests flow through tiers in order unless a tier is configured to approve and close.
ℹ
Conditions can be created at any tier and carry forward to all subsequent tiers. The final board sees everything flagged earlier and makes a holistic judgment. Conditions do not block escalation between tiers.

EA Specialist automation
Enable auto-approval
When all auto-approve conditions below are met and no hard gates or quality checks fail, the EA Specialist approves without human review. Start conservative and expand coverage as confidence grows.
Enable auto-rejection
When a submission triggers any hard gate rule below, the EA Specialist returns it to the submitter with the specific reason before it reaches any reviewer. Submitters fix the issue and resubmit. Use hard gates only for clear, unambiguous violations.

Step 5 of 7 · EA Specialist rules
Submitter assistance
Configure how the EA Specialist engages with submitters during form entry — before the request reaches any reviewer.
1 — Assistance mode *
🎓 Coach mode
EA Specialist surfaces findings inline as submitters fill the form — with options to resolve before submitting.
🚦 Gate mode
Evaluates on submission and returns pass/fail. No inline guidance during form entry.
2 — Findings to surface to submitters
Dynamically generated from your configuration in Steps 4.2, 5.1, and 5.2. Only finding types with rules behind them are shown. Grayed-out types have no rules configured yet.
From document template (Step 4.2)
Document section incomplete or off-topic
EA Specialist checks each required section against its guidance text. Flags sections that are empty, contain only placeholder text, or don't substantively address what the guidance asks for. Submitter sees the specific section name and what's missing.
Powered by: 6 required sections with guidance text defined in Step 4.2
Optional section absent on complex submission
For optional sections (e.g. Risks & assumptions), EA Specialist nudges the submitter if the submission appears complex or cross-domain but the section is missing.
Powered by: 1 optional section · quality check rule
From standards library (Step 5.1)
TRM conflict — product choice doesn't match approved usage
e.g. Redis used as message queue when only approved for caching. Submitter sees the specific TRM rule and alternative options to resolve it.
Powered by: Enterprise TRM Catalog 2026 · 47 products
Standards conflict — design violates a published standard
e.g. Direct database connection bypasses the API gateway. Submitter sees the specific standard and resolution options.
Powered by: Integration Standards v3.2, Architecture Guidelines
From form field configuration (Step 4.1)
Compliance gap — required field missing or below threshold
Business justification too short, architecture diagram not uploaded, or other required form fields empty or incomplete.
Powered by: Form field configuration · 6 required fields
